    Git Config

    Project Organization

    Moving forward we'll be creating a load of extra pages that go into detail about specific git commands. Before we get too much further along in this walkthrough let's create a commands directory inside our gitFundamentals directory.

    This folder can easily be created through the command line (if you're navigated to the gitFundamentals directory in your terminal) or through VS Code by clicking the New Folder option in your Explorer panel. This option appears when you hover over the explorer and is right next to the New File button we've used before.

     

    Create the Config.md

    Once you have created your commands folder, go ahead and add a Config.md file. This will be the file where we store information about the git config command.

    In this file we'll add a header and a bit of information about the command:

    # git config

When first setting up git, you'll need to work with the git config commands in order to set up your identity.

For example, your identity can be set up with the commands such as:

     

    We can also then add an example of the commands in use by using a markdown fenced code block. (Links to an external site.) Let's add the git config commands we used earlier underneath our above text:

    ```
git config --global user.name "Arnold Shortman"
git config --global user.email arnoldshortman202@gmail.com
```

     

    Underneath this code block we'll add a link to the git documentation about the first time git setup. Links in markdown are created with the text inside of square brackets and the link immediately following inside parentheses. For example:

    [Click me!](https://www.youtube.com/watch?v=HUBNt18RFbo)

    In your Config.md file, add the following code:

    For more information, view the [first time git setup](https://git-scm.com/book/en/v2/Getting-Started-First-Time-Git-Setup) documentation.

     

    Then finally, we'll add a few more items including Resources and a Back to Home link.

    ## Resources

- [Git Config Documentation](https://git-scm.com/docs/git-config)

---

[Back to home](../README.md)

     

    Let's review the items we've added:

    • A smaller header using the ## syntax.
    • A bullet pointed link, using the -  bullet point syntax along with the []() link syntax.
    • A horizontal rule using the --- syntax.
    • Another link, this time to a local file. The ../ syntax takes you back one directory from the commands folder to the root folder.

    When previewing your Config.md page in VS Code you should now be able to click the Back to home link and it should take you to a preview of the README.md file.

     

    Modify README.md

    Now that we have the Config.md file created, let's add a section to our README.md file where we link to all of our commands. First, open up the README file in VS Code.

    We'll add a new header underneath the current text and then start a bullet point list of links to the command pages.

    ## Git Commands
- [git config](./commands/Config.md)

    This should appear as a link in with a bullet point that takes you to the Config.md file.

    When viewing the preview in VS Code you should be able to click on the git config link and it will take you to the Config.md page. From there Back to home should return you to the README.md page.

     

    Commit your Code

    Now that we've got our general structure down we will want to move on to adding more commands. Before doing so, let's actually use git and commit our code.

    You can open a terminal in VS Code by using the ctrl + ` short cut or by selecting the Terminal tab at the top of VS Code's navigation options and selecting New Terminal. This will open a terminal in VS Code to the current directory.

    Inside a terminal, let's now run our first set of git commands.

    • git add .
      • Remember, this adds all current changes not tracked by the git repository to a group, staged to be committed.
    • git commit -m "Initial commit"
      • This is our first commit where we package up all staged changes inside a commit
      • The -m "Initial commit" is going to be the commit message. We've used Initial commit as this is the first commit so far. You can also do something like "Set up README and first command page" if you'd like to be more specific.

    And now we're good to move on! We've committed our changes and any changes we make now will be packaged into a new commit. If at any time we have an issue, we can revert back to this current commit. It's basically just a checkpoint!
